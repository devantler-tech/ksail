**Notes:**

- "Built-in" means the distribution includes this component by default
- K3s includes local-path-provisioner, metrics-server, and ServiceLB (load balancer) out of the box
- Talos CSI support is provider-dependent: Local Path Provisioner for Docker, Hetzner CSI Driver for Hetzner Cloud
- **LoadBalancer support by distribution** ÔÇö see [LoadBalancer Configuration](/configuration/loadbalancer/) for full details:
  - **Vanilla (Kind) on Docker**: Uses cloud-provider-kind (runs as external Docker container)
  - **K3s on Docker**: Uses built-in ServiceLB (Klipper-LB)
  - **Talos on Docker**: Uses MetalLB with default IP pool (172.18.255.200-172.18.255.250)
  - **Talos on Hetzner**: Uses Hetzner Cloud Load Balancer (cloud provider integration)
- **VCluster footnotes:**
  - ┬╣ CNI is managed internally by the vCluster control plane ÔÇö Vind configures networking within the Docker containers
  - ┬▓ CSI is managed internally by vCluster ÔÇö no separate CSI driver needed
  - ┬│ LoadBalancer is delegated to the host cluster by vCluster ÔÇö `spec.cluster.loadBalancer` has no effect on VCluster and KSail does not install any LoadBalancer controller
  - Ôü┤ Metrics Server is managed internally by vCluster
- **ÔüĚ KWOK Simulation**: Components are installed as API objects and their pods appear Running via KWOK's Stage simulation. They do not execute real workloads ÔÇö KWOK simulates pod lifecycle at the API level. See [KWOK Distribution](/distributions/kwok/) for details.
- **ÔüŞ KWOK Policy Engines**: Kyverno and Gatekeeper are not installed on KWOK. Both register global `MutatingWebhookConfigurations` that intercept all Kubernetes API requests. On KWOK, no real pod serves the webhook endpoint, so every webhook call times out ÔÇö breaking all subsequent Helm installs. KSail silently skips policy engine installation when `spec.cluster.distribution: KWOK` is set and emits a warning at cluster creation time.
- **Ôü╣ KWOK Flux**: Flux is not installed on KWOK. KSail skips Flux installation and configuration entirely (with a warning) because Flux cannot function on KWOK. GitOps reconciliation is not functional on KWOK, including with ArgoCD, because controller pods are only simulated and cannot sync resources. Use a non-KWOK distribution for real GitOps syncing, or use `ksail workload apply` on KWOK for non-GitOps manifest application.
- **┬╣Ôü░ EKS (Planned)**: Full component installer support for EKS is in progress. `ksail cluster init` is available, but `ksail cluster create` is not yet functional. See [EKS Distribution](/distributions/eks/) for details.
- **┬╣┬╣ KWOK CNI**: Cilium and Calico are not installed on KWOK. KWOK runs simulated pods with no real network dataplane, so CNI plugins are never functional and are always skipped. If a non-default CNI is configured in `ksail.yaml` for a KWOK cluster, KSail emits a warning and skips installation. The `spec.cluster.cni` setting has no effect on KWOK clusters.
- **┬╣┬▓ KWOK CSI and cert-manager**: Local Path Provisioner (CSI) and cert-manager are not installed on KWOK. The Local Path Provisioner Deployment pod and cert-manager webhook pods require real container processes that KWOK does not provide, so readiness never becomes true and installation would time out. KSail skips installation of these components when `spec.cluster.distribution: KWOK` is set and emits a warning at cluster creation time.

## Update Behavior

The `ksail cluster update` command applies configuration changes to a running cluster.
Changes are classified by impact: **in-place** (no disruption), **reboot-required** (node restart needed), or **recreate-required** (full cluster recreation).

| Change                   | Vanilla (Kind) | K3s (K3d) | Talos    | VCluster (Vind) | KWOK (kwokctl) | EKS        |
| ------------------------ | -------------- | --------- | -------- | --------------- | -------------- | ---------- |
| Distribution             | Recreate       | Recreate  | Recreate | Recreate        | Recreate       | ­čÜž┬╣Ôü░      |
| Provider                 | Recreate       | Recreate  | Recreate | Recreate        | Recreate       | ­čÜž┬╣Ôü░      |
| CNI                      | In-place       | In-place  | In-place | N/A             | N/A┬╣┬╣          | ­čÜž┬╣Ôü░      |
| CSI                      | N/AÔüÁ           | In-place  | In-place | N/A             | ÔŁî┬╣┬▓           | ­čÜž┬╣Ôü░      |
| Metrics Server           | In-place       | In-place  | In-place | N/A             | SimÔüĚ           | ­čÜž┬╣Ôü░      |
| Load Balancer            | In-place       | In-place  | N/A┬╣     | N/A             | SimÔüĚ           | ­čÜž┬╣Ôü░      |
| cert-manager             | In-place       | In-place  | In-place | In-place        | ÔŁî┬╣┬▓           | ­čÜž┬╣Ôü░      |
| Policy Engine            | In-place       | In-place  | In-place | In-place        | ÔŁîÔüŞ            | ­čÜž┬╣Ôü░      |
| GitOps Engine            | In-place       | In-place  | In-place | In-place        | SimÔüĚ           | ­čÜž┬╣Ôü░      |

| Mirrors Dir              | Recreate       | N/A       | N/A      | N/A             | N/A            | ÔŁî         |
| Control Planes           | Recreate       | Recreate  | In-place | Recreate        | Recreate       | ­čÜž┬╣Ôü░      |
| Workers                  | Recreate       | In-place  | In-place | N/A             | N/A            | ­čÜž┬╣Ôü░      |
| Hetzner Server Type (CP) | N/A            | N/A       | Recreate | N/A             | N/A            | N/A        |

**Notes:**

- "In-place" changes are applied without cluster downtime via Helm or provider APIs
- "Recreate" changes trigger a cluster recreation. `ksail cluster update` will prompt before proceeding; use `--force` (or `-y` / `--yes`) to skip the prompt for non-interactive runs, or run `ksail cluster delete && ksail cluster create` manually.
- If no changes are detected, `ksail cluster update` exits immediately with no modifications (idempotent no-op)
- Use `ksail cluster update --dry-run` to preview changes before applying; add `--output json` for machine-readable diff output (suitable for CI gating or [MCP tools](/mcp/))
- Talos supports the broadest set of in-place updates, including node scaling for both control-plane and worker nodes across all providers ÔÇö including Omni (via declarative cluster template sync)
- Kind does not support any structural node changes after creation
- VCluster (Vind) runs control-plane and optional worker nodes as Docker containers ÔÇö CNI, CSI, and metrics-server are managed by the vCluster Helm chart and are N/A for update operations; LoadBalancer services are provided by the host cluster, and `spec.cluster.loadBalancer` is ignored for VCluster and will not trigger a cluster update
- **┬╣ Load Balancer for Talos**: For Talos, the provider determines which LoadBalancer implementation is used (MetalLB on Docker, Hetzner Cloud Controller Manager on Hetzner). The `spec.cluster.loadBalancer` setting controls whether KSail installs LoadBalancer support. See [FAQ](/faq/#which-distributions-support-loadbalancer-services) for details.
- **ÔüÁ CSI for Vanilla (Kind)**: Kind bundles local-path-provisioner by default. KSail's detector reports it as `CSIEnabled` but cannot distinguish Kind's bundled CSI from a KSail-installed CSI driver, so CSI comparison is skipped entirely during `ksail cluster update`. To change CSI settings on a Vanilla cluster, recreate it with `ksail cluster delete && ksail cluster create`.
- **ÔüÂ Image Verification for Vanilla (Kind)**: Uses the containerd `io.containerd.image-verifier.v1.bindir` plugin (requires containerd 2.x / Kind `v0.31.0+` / `kindest/node:v1.35.1+`). Verifier binaries (e.g., Cosign, Notation) must be pre-installed in a custom Kind node image at `/opt/image-verifier/bin`. See [Vanilla Image Verification](/distributions/vanilla/#image-verification).
