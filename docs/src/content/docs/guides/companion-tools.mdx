KSail manages **cluster provisioning and GitOps bootstrapping**. Inner-loop tools handle the **build-watch-deploy** cycle, hot-reload, or local↔remote traffic bridging. They are complementary—not competing—layers that work together seamlessly.

| Layer | Responsibility | Tool |
|-------|---------------|------|
| Cluster lifecycle | Create, configure, and delete clusters | **KSail** |
| GitOps bootstrap | Install Flux/ArgoCD, push manifests | **KSail** |
| Manifest watch & apply | Watch files, auto-apply Kustomize layers | **KSail** (`ksail workload watch`) |
| Build & deploy loop | Watch files, rebuild images, hot-reload | **Tilt**, **Skaffold**, or **DevSpace** |
| Local↔remote bridging | Bridge local processes to in-cluster pods | **Telepresence** or **mirrord** |

## KSail + Tilt

[Tilt](https://tilt.dev/) watches your source files, rebuilds container images, and applies updated manifests to a running cluster. KSail creates and configures that cluster.

### Workflow

1. **Create the cluster with KSail**

   ```bash
   ksail cluster init --name dev --distribution K3s --cni Cilium
   ksail cluster create
   ```

2. **Write a `Tiltfile`** that targets the KSail-managed kubeconfig context.

   KSail sets the kubeconfig context automatically after `ksail cluster create`. The context name depends on the distribution:

   | Distribution | Context name |
   |-------------|-------------|
   | Vanilla (Kind) | `kind-<name>` |
   | K3s (K3d) | `k3d-<name>` |
   | Talos | `admin@<name>` |
   | VCluster | `vcluster-docker_<name>` |

   ```python
   # Tiltfile
   allow_k8s_contexts("k3d-dev")

   docker_build("my-app", ".")

   k8s_yaml("k8s/deployment.yaml")
   k8s_resource("my-app", port_forwards="8080:80")
   ```

3. **Start Tilt**

   ```bash
   tilt up
   ```

   Tilt connects to the KSail-managed cluster, rebuilds the image on every file save, and applies the updated manifest.

4. **Tear down**

   ```bash
   tilt down
   ksail cluster delete
   ```

### Tips

- Run `ksail cluster info` to confirm the cluster is healthy before starting Tilt.
- If you use a local registry (`--local-registry localhost:5050`), configure Tilt to push there:

  ```python
  # Tiltfile
  default_registry("localhost:5050")
  ```

## KSail + DevSpace

[DevSpace](https://devspace.sh/) provides hot-reload for interpreted languages (Python, Node.js, Ruby) without rebuilding container images. It syncs file changes directly into running containers and can restart processes automatically. KSail creates and configures the cluster; DevSpace handles the inner-loop.

### Workflow

1. **Create the cluster with KSail**

   ```bash
   ksail cluster init --name dev --distribution K3s --cni Cilium
   ksail cluster create
   ```

2. **Initialize DevSpace** in your project directory.

   ```bash
   devspace init
   ```

   This creates a `devspace.yaml`. Configure it to use the KSail-managed kubeconfig context:

   | Distribution | Context name |
   |-------------|-------------|
   | Vanilla (Kind) | `kind-<name>` |
   | K3s (K3d) | `k3d-<name>` |
   | Talos | `admin@<name>` |
   | VCluster | `vcluster-docker_<name>` |

   ```yaml
   # devspace.yaml
   version: v2beta1
   name: my-app

   pipelines:
     dev:
       run: |-
         run_dependencies --all
         ensure_pull_secrets --all
         run_pipelines deploy
         start_dev --all

   dev:
     my-app:
       imageSelector: my-app
       sync:
         - path: ./src:/app/src
       terminal: {}
       ports:
         - port: "8080:80"
   ```

3. **Start DevSpace dev mode**

   ```bash
   devspace dev --kube-context k3d-dev
   ```

   DevSpace syncs local file changes into the running container and opens a terminal session. For interpreted languages, changes appear instantly without image rebuilds.

4. **Tear down**

   ```bash
   devspace purge
   ksail cluster delete
   ```

### Tips

- Use `devspace dev --kube-context <context>` to target the KSail-managed cluster explicitly.
- DevSpace file sync is ideal for interpreted languages. For compiled languages, use `devspace.yaml` build pipelines or pair with Tilt.
- Run `ksail cluster info` to verify the cluster is healthy before starting DevSpace.

## KSail + Telepresence

[Telepresence](https://www.telepresence.io/) bridges your local machine to the cluster network, letting you run a service locally while it receives traffic from in-cluster pods. This is useful for debugging services in Talos or Hetzner KSail clusters without image rebuilds.

### Workflow

1. **Create a cluster and deploy your application**

   ```bash
   ksail cluster init --name dev --distribution K3s --cni Cilium
   ksail cluster create

   # Deploy your workload
   kubectl create deployment my-app --image=my-app:latest
   kubectl expose deployment my-app --port=80
   ```

2. **Connect Telepresence to the cluster**

   ```bash
   telepresence connect --context k3d-dev
   ```

   Telepresence installs a traffic manager in the cluster and creates a network tunnel from your local machine.

3. **Intercept traffic for your service**

   ```bash
   telepresence intercept my-app --port 8080:80
   ```

   All traffic destined for the `my-app` service in the cluster is now routed to `localhost:8080`. Run your local process on that port:

   ```bash
   # Run your service locally — it receives cluster traffic
   go run ./cmd/server --port 8080
   ```

4. **Tear down**

   ```bash
   telepresence leave my-app
   telepresence quit
   ksail cluster delete
   ```

### Tips

- Telepresence intercepts work with any KSail distribution (Vanilla, K3s, Talos, VCluster).
- Use `telepresence list` to see available services for interception.
- For selective routing, use `--http-header` to intercept only requests matching specific headers (useful in shared clusters).

## KSail + Skaffold

[Skaffold](https://skaffold.dev/) automates the build-tag-deploy pipeline. KSail provides the cluster; Skaffold handles the rest.

### Workflow

1. **Create the cluster with KSail**

   ```bash
   ksail cluster init --name dev --distribution Vanilla --cni Cilium
   ksail cluster create
   ```

2. **Create `skaffold.yaml`** pointing at the KSail cluster context.

   ```yaml
   # skaffold.yaml
   apiVersion: skaffold/v4beta11
   kind: Config
   metadata:
     name: my-app
   build:
     artifacts:
       - image: my-app
   manifests:
     rawYaml:
       - k8s/deployment.yaml
   deploy:
     kubeContext: kind-dev
     kubectl: {}
   ```

3. **Run Skaffold in dev mode**

   ```bash
   skaffold dev
   ```

   Skaffold watches for file changes, rebuilds the image, and redeploys to the KSail-managed cluster.

4. **Tear down**

   ```bash
   # Ctrl+C to stop Skaffold (it cleans up deployed resources)
   ksail cluster delete
   ```

### Tips

- Use `skaffold dev --port-forward` for automatic port forwarding.
- If you enabled a local registry in KSail, point Skaffold at it:

  ```yaml
  # skaffold.yaml (build section)
  build:
    local:
      push: true
    artifacts:
      - image: localhost:5050/my-app
  ```

## KSail + mirrord

[mirrord](https://mirrord.dev/) lets you run a local process as if it were inside the cluster by mirroring or stealing traffic from a remote pod. KSail sets up the cluster; mirrord connects your local process to it.

### Workflow

1. **Create a cluster and deploy your application**

   ```bash
   ksail cluster init --name dev --distribution K3s --cni Cilium
   ksail cluster create

   # Deploy your workload
   kubectl create deployment my-app --image=my-app:latest
   kubectl expose deployment my-app --port=80
   ```

2. **Create a mirrord config** (optional—mirrord works without one).

   ```json
   {
     "target": {
       "path": "deployment/my-app",
       "namespace": "default"
     },
     "feature": {
       "network": {
         "incoming": "mirror"
       },
       "fs": "read"
     }
   }
   ```

   Save this as `.mirrord/mirrord.json` in your project root.

3. **Run your process through mirrord**

   ```bash
   mirrord exec --config .mirrord/mirrord.json -- ./my-app
   ```

   Your local process receives traffic destined for `deployment/my-app` inside the KSail-managed cluster, while reading remote files and environment variables from the pod.

4. **Tear down**

   ```bash
   # Stop the mirrord session (Ctrl+C)
   ksail cluster delete
   ```

### Tips

- mirrord also integrates as a VS Code and JetBrains plugin—no CLI needed.
- Use `"incoming": "steal"` instead of `"mirror"` to redirect all traffic to your local process rather than duplicating it.

## When to Use Which Tool

`ksail workload watch` handles manifest-level iteration — it watches your `k8s/` directory and auto-applies changes via `kubectl apply`. Companion tools cover different layers:

| Scenario | Recommended tool |
|----------|-----------------|
| Editing Kubernetes manifests (YAML/Kustomize) | `ksail workload watch` |
| Hot-reloading interpreted code (Python, Node.js) without image rebuilds | **DevSpace** |
| Rebuilding container images on file save | **Tilt** or **Skaffold** |
| Debugging a service locally with live cluster traffic | **Telepresence** |
| Mirroring pod traffic to a local process | **mirrord** |

You can combine `ksail workload watch` with any companion tool — they operate at different layers and do not conflict.

## Combining Multiple Tools

You can layer these tools together. For example, use Tilt for the build-deploy loop and mirrord for a specific service you want to debug locally:

```bash
# Terminal 1 — cluster + Tilt
ksail cluster create
tilt up

# Terminal 2 — mirrord for live debugging
mirrord exec --target deployment/my-app -- dlv debug ./cmd/server
```
