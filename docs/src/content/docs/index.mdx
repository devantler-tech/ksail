## Why KSail?

Standing up Kubernetes the real way means juggling kind or k3d, helm, kubectl, a GitOps engine, SOPS, and a pile of glue scripts — and every distribution and environment does it a little differently.

KSail collapses all of that into **one binary and one declarative config**. The same workflow — `init` → `create` → `push` → `reconcile` — works across Vanilla, K3s, Talos, VCluster, KWOK, and EKS, and from your laptop to production. No tool sprawl, and no lock-in: KSail generates native `kind.yaml`, Talos patches, and friends that you can run with or without it.

## Quick Start

Scaffold a project, create a cluster, and deploy a workload — then follow the full
[Quickstart](/start/quickstart/) and [Deliver with GitOps](/start/deliver-with-gitops/) tutorials.

<img class="demo-recording" src="/demos/cluster-init.gif" alt="ksail cluster init scaffolds ksail.yaml and kind.yaml, cat shows the minimal declarative config, and re-running with --distribution K3s switches the distribution" />

```bash
# 1. Initialize a new project (pick any distribution)
ksail cluster init --name my-cluster --distribution Vanilla

# 2. Create and start the cluster
ksail cluster create

# 3. Add manifests under k8s/, then deploy
ksail workload apply -k ./k8s

# 4. Open K9s to explore
ksail cluster connect
```

**Prerequisites:** KSail runs on Linux, macOS, and Windows (WSL2). Local clusters need
[Docker](/installation/); the [Kubernetes (Nested) provider](/providers/kubernetes/) runs inside an
existing cluster instead. See the [Support Matrix](/support-matrix/) for every distribution × provider
combination.
