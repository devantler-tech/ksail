## Learning Kubernetes

For developers new to Kubernetes who want to explore concepts and experiment with different configurations.

### Recommended Setup

```bash
ksail cluster init \
  --name learning \
  --distribution Vanilla \
  --cni Cilium \
  --gitops-engine None
```

Vanilla provides standard upstream Kubernetes, Cilium adds modern networking with observability, and skipping GitOps enables a direct apply workflow for immediate feedback.

### Workflow

1. **Create cluster**

   ```bash
   ksail cluster create
   ```

2. **Apply manifests directly**

   ```bash
   ksail workload apply -f my-deployment.yaml
   ksail workload get pods
   ```

3. **Experiment with changes**

   ```bash
   ksail workload apply -f updated-deployment.yaml
   ksail workload logs deployment/my-app
   ```

4. **Clean up**

   ```bash
   ksail cluster delete
   ```

### Tips

- Use `ksail workload gen` to generate example manifests
- Use `ksail workload explain <resource>` to learn about Kubernetes resources
- Use `ksail workload watch` to watch for file changes and auto-apply; it scopes `kubectl apply` to the nearest directory containing a `kustomization.yaml` for faster iteration
- Use `ksail cluster connect` to open K9s for interactive exploration

## Iterating on Applications

For developers building and testing applications locally before deploying to staging or production.

### Recommended Setup

```bash
ksail cluster init \
  --name dev \
  --distribution K3s \
  --cni Cilium \
  --csi Enabled \
  --gitops-engine Flux \
  --local-registry localhost:5050
```

K3s starts faster with lower resource usage, Cilium provides network policies and observability, Local Path Provisioner enables persistent volumes for stateful apps, Flux mirrors production GitOps patterns, and a local registry avoids pushing images to a remote registry during iteration.

### Workflow

1. **Create cluster**

   ```bash
   ksail cluster create
   ```

2. **Build and push image**

   ```bash
   docker build -t localhost:5050/my-app:dev .
   docker push localhost:5050/my-app:dev
   ```

3. **Update manifests**

   ```yaml
   # k8s/deployment.yaml
   image: localhost:5050/my-app:dev
   ```

4. **Deploy via GitOps**

   ```bash
   ksail workload push
   ksail workload reconcile
   ```

### Tips

- Use `ksail workload logs -f deployment/my-app` for live log streaming
- Use `ksail workload exec deployment/my-app -- /bin/sh` for debugging
- Keep terminal running with `ksail cluster info` to monitor cluster health
- Pair KSail with **[Tilt, Skaffold, or mirrord](/guides/companion-tools/)** to automate the build-deploy loop or mirror live traffic to your local process

## Testing in CI/CD

For automated testing in CI/CD pipelines where reproducibility and speed matter.

### Recommended Setup

```yaml
# ksail.yaml
apiVersion: ksail.io/v1alpha1
kind: Cluster
spec:
  cluster:
    distribution: K3s
    cni: Cilium
    gitOpsEngine: Flux
    localRegistry:
      registry: localhost:5050
  workload:
    sourceDirectory: k8s
```

K3s starts fastest in CI, Cilium enforces network policies, Flux tests the actual deployment workflow, and declarative config ensures reproducibility across runs.

### CI Pipeline Example

```yaml
# .github/workflows/test.yaml
name: Integration Tests

on:
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install KSail
        run: |
          curl -sSL https://github.com/devantler-tech/ksail/releases/latest/download/ksail_linux_amd64 -o ksail
          chmod +x ksail
          sudo mv ksail /usr/local/bin/

      - name: Create cluster
        run: ksail cluster create

      - name: Build and push
        run: |
          docker build -t localhost:5050/my-app:${{ github.sha }} .
          docker push localhost:5050/my-app:${{ github.sha }}

      - name: Deploy
        run: |
          sed -i "s|image:.*|image: localhost:5050/my-app:${{ github.sha }}|" k8s/deployment.yaml
          ksail workload push
          ksail workload reconcile

      - name: Run tests
        run: |
          # Wait for deployment
          ksail workload wait deployment/my-app --for=condition=available

          # Run integration tests
          npm run test:integration

      - name: Cleanup
        if: always()
        run: ksail cluster delete
```

You can add `--ttl` as a best-effort safety net. When backgrounding the create step, include an explicit readiness wait so subsequent steps don't start before the cluster is ready:

```yaml
      - name: Create cluster (auto-destroys after 30m)
        run: ksail cluster create --ttl 30m &

      - name: Wait for cluster readiness
        run: kubectl wait --for=condition=Ready node --all --timeout=120s
```

> **Note:** TTL deletion only runs while the `ksail cluster create` process is alive. If the CI runner is cancelled or the job terminates, the background process is killed and the cluster won't be deleted by TTL. Keep the `if: always()` cleanup step as the primary safeguard.

### Tips

- Use `--timeout` flags to handle slow CI runners
- Cache Docker layers for faster builds
- Use matrix builds to test across multiple Kubernetes versions/distributions
- Use `--ttl` as a best-effort supplement on ephemeral runners — the cluster auto-destroys when the TTL elapses provided the `ksail cluster create` process stays alive
- Consider using `ksail cluster start` and `ksail cluster stop` if tests can share a cluster

## Configuration Comparison

| Aspect           | Learning         | Development      | CI/CD            |
|------------------|------------------|------------------|------------------|
| Distribution     | Vanilla          | K3s              | K3s              |
| CNI              | Cilium           | Cilium           | Cilium           |
| CSI              | None             | Local Path Provisioner | None        |
| GitOps           | None             | Flux             | Flux             |
| Local Registry   | Optional         | Yes              | Yes              |
| Declarative Config | Optional       | Recommended      | Required         |
| Cluster Lifetime | Session          | Days/Weeks       | Minutes (or `--ttl`) |
